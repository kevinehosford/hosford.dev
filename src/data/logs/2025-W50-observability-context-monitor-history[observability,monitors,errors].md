### Overview
Built an isomorphic observability context so `user_id`, `org_id`, `request_id`, and `trace_id` ride along through logs, traces, and Sentry ŌĆö point of all this being to make support triage actually possible. Decoupled monitor history from the list fetch so opening Monitors stops blocking on history, and consolidated the error boundary mess into a single shape.

### Highlights
- [#6488](https://github.com/axiomhq/app/pull/6488) ŌĆö Added `lib/observability-context` to manage `user_id` / `org_id` / `request_id` / `trace_id` across server and browser, syncing with Sentry (user/tags) and clearing on sign-out. The `traces` middleware injects `trace_id` after extracting context and stamping `x-axiom-trace-id`; the logging middleware picks the rest up. Now we can pivot from a Sentry event to logs to traces for the same request without doing string surgery.
- [#6463](https://github.com/axiomhq/app/pull/6463) ŌĆö Split monitor history off the list query. Added `getMonitorsQuery`, `getMonitorsHistoryQuery`, `useMonitorsWithOptions`, and `useMonitorsByState`, then ripped out `MonitorsStore` / `useMonitors` / `getMonitorsWithHistoryQuery`. Consumers now show monitors immediately and let history populate in the background via `QueryKeys.monitorsHistory()`.
- [#6521](https://github.com/axiomhq/app/pull/6521) ŌĆö Cleaned up three related messes at once. Server functions were swallowing the real atlas/core error message, so added `HTTPError` and `RoutePermissionError` serialization adapters in `lib/error-adapter.ts` and registered them in `start.ts`. The error boundary surface was full of duplication, so merged everything into one. And the "invalid token on initial load via server fn" case (mostly hitting devs) now inspects the serialized error for the atlas message and triggers the sign-out flow with `reset=true` so the dead token gets cleaned up.
